# URL Shortener (Go) — учебный проект

Простой сервис сокращения ссылок на Go.  
Этот репозиторий специально сделан как старт для новичка.

## Что уже умеет проект

- Принимает длинную ссылку и создает короткую.
- Хранит данные в памяти (`map`), без базы данных.
- Короткие коды идут по порядку: `1`, `2`, `3`, ...
- Делает редирект с короткой ссылки на исходную.

Важно: после перезапуска сервера все ссылки исчезают (это нормально для учебной версии).

## Быстрый старт (пошагово)

### Шаг 1. Установи Go

Нужна версия Go `1.22+`.

Проверь:

```bash
go version
```

### Шаг 2. Перейди в папку проекта

```bash
cd c:\Users\ubadm\golangprojects\url_shortener
```

### Шаг 3. Запусти сервер

```bash
go run .
```

После запуска сервер доступен по адресу:
- `http://localhost:8080`

### Шаг 4. Проверь, что сервер жив

Открой в браузере:
- `http://localhost:8080`

Должен увидеть текст:
- `URL Shortener is running`

### Шаг 5. Создай короткую ссылку

Через PowerShell:

```powershell
curl -X POST "http://localhost:8080/shorten?url=https://example.com"
```

Пример ответа:

```text
short url: http://localhost:8080/1
```

### Шаг 6. Проверь редирект

Открой в браузере:
- `http://localhost:8080/1`

Браузер должен перенаправить тебя на `https://example.com`.

## Как устроен текущий код (простыми словами)

- `shortener.links` — словарь: `короткий_код -> оригинальная_ссылка`.
- `shortener.next` — счетчик следующего кода.
- `POST /shorten` — читает `url` из query-параметра и сохраняет ссылку.
- `GET /{code}` — ищет ссылку по коду и делает `redirect`.
- `GET /` — простой health-check.

## План развития до “большого” проекта (для новичка, по шагам)

Ниже дорожная карта. Иди строго по этапам, не перескакивай через 3-4 пункта.

## Этап 0. Закрепить базу Go и HTTP

Цель: уверенно понимать текущий код.

Сделай:
1. Поменяй текст ответа на `/`.
2. Добавь проверку пустого `url`.
3. Добавь логирование создания короткой ссылки.

Готово, если:
- сам можешь объяснить каждую функцию в `main.go`.

## Этап 1. Разделить код по папкам

Цель: перейти от “все в одном файле” к нормальной структуре.

Сделай:
1. Вынеси запуск сервера в `cmd/api/main.go`.
2. Создай `internal/handler` для HTTP-обработчиков.
3. Создай `internal/service` для бизнес-логики.
4. Создай `internal/repository` для хранилища.

Готово, если:
- код все еще запускается командой `go run ./cmd/api`.

## Этап 2. Добавить тесты

Цель: научиться безопасно менять код.

Сделай:
1. Unit-тест для функции проверки URL.
2. Unit-тест для создания короткого кода.
3. HTTP-тест для `POST /shorten`.
4. HTTP-тест для `GET /{code}`.

Готово, если:
- `go test ./...` проходит стабильно.

## Этап 3. Добавить PostgreSQL

Цель: чтобы ссылки не терялись после перезапуска.

Сделай:
1. Подними Postgres через `docker-compose`.
2. Создай таблицу `links` (`id`, `code`, `original_url`, `created_at`).
3. Перепиши repository на работу с Postgres.
4. Добавь миграции.

Готово, если:
- перезапустил сервер, а старые ссылки остались.

## Этап 4. Сделать API “взрослым”

Цель: нормальный формат для фронта и клиентов.

Сделай:
1. Перейди на JSON-ответы.
2. Сделай единый формат ошибок.
3. Добавь endpoint для получения информации о ссылке.
4. Добавь удаление ссылки.

Готово, если:
- API документировано и понятно без чтения кода.

## Этап 5. Документация API (OpenAPI/Swagger)

Цель: удобно тестировать и отдавать API другим.

Сделай:
1. Описать эндпоинты в OpenAPI.
2. Подключить Swagger UI.
3. Обновить README с примерами запросов/ответов.

Готово, если:
- можно открыть Swagger и попробовать API в браузере.

## Этап 6. Надежность и наблюдаемость

Цель: подготовка к реальной эксплуатации.

Сделай:
1. Таймауты у HTTP-сервера.
2. Graceful shutdown.
3. Структурированные логи.
4. `/health/live` и `/health/ready`.
5. Метрики (`/metrics`) + Prometheus.

Готово, если:
- при ошибках и нагрузке понятно, что происходит.

## Этап 7. CI/CD и деплой

Цель: автоматическая проверка и удобный релиз.

Сделай:
1. Dockerfile.
2. CI (тесты + линтер + сборка).
3. Деплой на VPS/облако.
4. Reverse proxy (Nginx/Caddy), домен и TLS.

Готово, если:
- одно изменение в `main` автоматически проверяется и выкатывается.

## Частые ошибки новичка

- Пытаться сразу добавить “все и сразу” (БД, auth, Docker, метрики).
- Не писать тесты перед большим рефакторингом.
- Смешивать бизнес-логику и HTTP-код в одном месте.
- Делать сложную генерацию кодов слишком рано.

Лучший путь: маленький шаг -> тест -> следующий шаг.
